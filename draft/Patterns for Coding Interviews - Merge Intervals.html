Patterns for Coding Problems - Merge Intervals

Problem

  1. Merge Intervals
  2. Insert Interval
  3. Interval List Intersections

Approach

Take Merge Intervals as an example, given a set of arrays representing intervals, determine if there are overlaps exist within it. If so, merge those overlapping intervals together. Some cases are shown below:

Input: [[1,3],[2,6],[8,10],[15,18]]
Output: [[1,6],[8,10],[15,18]]
Explanation: Since intervals [1,3] and [2,6] overlaps, merge them into [1,6].
Input: [[1,4],[4,5]]
Output: [[1,5]]
Explanation: Intervals [1,4] and [4,5] are considered overlapping.

The most intuitive way is to compare each interval with every other intervals to checkout whether overlaps exist, which is a brute force approach. The method would return the time complexity with $O(n^2)$, as all intervals might be mutually overlapping in the worst case. Besides, the space complexity will be $O(n^2)$ as it would need a space to memorize those edges for every pair of intervals which have been visited.

The more efficient way is to sort those intervals by their starting values, then observe the pattern within the set. The image below will show how sorted intervals would be looked like:

Within the figure above, it can be found that the starting point of each merged interval is always the smallest value of those overlapping intervals, and the ending point is always the largest value of them. Now, we could come up with the idea of the key process:

for (int[] curInterval : intervals) {
  // no overlapping with the last interval
  if (mergedIntervals.isEmpty() || curArr[0] > mergedInterval.getLast()[1]) {
    mergedIntervals.add(curInterval);
  }
  // overlapping with the last interval
  else {
    mergedIntervals.getLast()[1] = Math.max(curInterval[1], mergedIntervals.getLast()[1]);
  }
}

The code above has shown situations that current interval has or has no overlap with previous interval by comparing the starting point of current interval and the last point of previous interval. If overlaps do not exist, the current interval will be inserted into the result set of intervals, otherwise, checkout which one is larger within the ending point of current interval and previous interval. Combine with the sorting function we have mentioned, the complete code would be like:

class Solution {
  public int[][] merge(int[][] intervals) {
    if (intervals == null || intervals.length < 2) {
      return intervals;
    }

    List<int[]> mergedIntervals = new ArrayList<>();
    Arrays.sort(intervals, (a, b) -> Integer.compare(a[0], b[0]));

    int start = intervals[0][0];
    int end = intervals[0][1];

    for (int[] interval : intervals) {
      if (interval[0] <= end) {
        end = Math.max(interval[1], end);
      } else {
        mergedIntervals.add(new int[] {start, end});
        start = interval[0];
        end = interval[1];
      }
    }

    mergedIntervals.add(new int[]{start, end});

    return mergedIntervals.toArray(new int[0][]);
  }
}

The approach shown above could reduce the time complexity to $O(nlogn)$, which dominated by the sorting function, and the space complexity would be as $O(1)$ as intervals could be sorted in place and without extra space.

Conclusion

Besides Merge Intervals, other problems related to intervals could also be solved as the approach like: sorting → observe pattern within sorted intervals (think of situations that intervals has or has no overlapping).

Take Interval List Intersections for instance, the problem requires us to find out intersections between both sets of sorted intervals. As intervals has been sorted, we can save the time for sorting and focus on searching the pattern within those intervals. So we further listing out situations that both intervals has or has no intersection, then we can find out that the starting point of the result interval will be max(a_start, b_start) and the ending point will be min(a_end, b_end), which can lead us to the key part of solving this kind of problems.
