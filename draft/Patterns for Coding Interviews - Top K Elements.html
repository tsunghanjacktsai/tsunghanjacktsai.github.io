Patterns for Coding Interviews - Top ‘K’ Elements

Introduction

The top K elements pattern is a technique that aims to return a given number of the most frequent/largest/smallest elements in a given set.

The key data structure of solving top K elements problems is a heap. As the feature of max heap and min heap, we can always get the largest and smallest elements efficiently, i.e., $O(logN)$.

Approach

Let’s see some examples. We’ll first look at a LeetCode medium problem — Kth Largest Element in an Array. It requires us to find a given number of the largest element in a set of integers.

Here are some examples:

Input: [3,2,1,5,6,4] and k = 2
Output: 5
Input: [3,2,3,1,2,4,5,5,6] and k = 4
Output: 4

A brute-force solution is to simply sort the array and find out the element at the Kth place. However, this method will result in $O(NlogN)$ due to the sorting function of the algorithm.

So can we improve the time complexity of this algorithm?

Definitely. Instead of sorting and searching separately, we can maintain a K size heap, which enables storing and sorting the visited elements at the same time.

But what kind of heap should we use? Basically min heap and max heap can both achieve the correct solution. Here we’ll explain how we can solve this problem by a min heap.

After deciding the data structure, we can now enter the main process. The idea of solving this problem by the top K elements pattern is to keep the given size of our min heap.

Since the root element will always be the smallest within a min heap, if the size of a heap is larger than K, the root element should be discarded as it is the (K+1)th largest element so far.

As a result, the algorithm process will be like:

  1. Traverse the array elements and put them into a K size min heap. If the heap size is larger than K, pop out the root element. Repeat this process until the end of the array.
  2. Return the root element of the min heap as it is the Kth largest element of the whole array.

A visualization of the algorithm is provided below:

Now we can easily write the code implementation:

class Solution {
  public int findKthLargest(int[] nums, int k) {
    // Create a min heap
    PriorityQueue<Integer> minHeap = new PriorityQueue<>((a, b) -> a - b);

    // Insert the array elements into the min heap.
    for (int i = 0; i < nums.length; i++) {
      minHeap.add(nums[i]);
      // Pop out the root element if the min heap size is larger than k
      if (minHeap.size() > k) {
        minHeap.poll();
      }
    }

    return minHeap.peek();
  }
}

By applying the top K elements pattern, we can improve the time complexity of the solution to $O(NlogK)$, in which the extraction of the root element is $O(logK)$, and we need to traverse the whole array.

Next, we’re gonna look at another LeetCode medium problem — K Closest Points to Origin. This is a bit different from the above problem which required us to find the K largest elements, whereas this one is asking for the K smallest elements.

Here are some examples:

Input: points = [[1,3],[-2,2]], K = 1
Output: [[-2,2]]
Explanation: 
The distance between (1, 3) and the origin is sqrt(10).
The distance between (-2, 2) and the origin is sqrt(8).
Since sqrt(8) < sqrt(10), (-2, 2) is closer to the origin.
We only want the closest K = 1 points from the origin, so the answer is just [[-2,2]].
Input: points = [[3,3],[5,-1],[-2,4]], K = 2
Output: [[3,3],[-2,4]]

Like we did before, we all know that a K size heap is able to solve the problem, however, which kind of heap should we use?

Again, for top k elements problems, we can always solve them by either max heap or min heap. There is just a bit different between how they handle a problem. For this one, we’ll use a max heap to show you how to deal with the K smallest problems.

The way we define a max heap also needs to be careful. Instead of comparing each integer directly, this time we’re going to compare the distance to the origin between each point.

So our max heap should be defined as:

PriorityQueue<int[]> maxHeap = new PriorityQueue<>((a, b) -> (b[0] * b[0] + b[1] * b[1]) - (a[0] * a[0] + a[1] * a[1]))

(b[0] * b[0] + b[1] * b[1]) - (a[0] * a[0] + a[1] * a[1]) means the comparison between the distance from b to the origin and the distance from a to the origin.

The rest of the solution will be the same as the last problem, which we’ll maintain a K size max heap. Elements inside this heap will be the K closest points to the origin. So the process will be like:

  1. Traverse and insert those points into the max heap. If the max heap size exceeds K, pop out the root element as it is the (K+1)th closest points to the origin so far which we don’t need it.
  2. Return the final max heap as the result.

Thus we can now implement the solution:

class Solution {
  public int[][] kClosest(int[][] points, int K) {
    // Max heap (comparess distance)
    PriorityQueue<int[]> maxHeap = new PriorityQueue<>((a, b) -> (b[0] * b[0] + b[1] * b[1]) - (a[0] * a[0] + a[1] * a[1]));

    // Traverse and insert points to the max heap
    for (int[] point : points) {
      maxHeap.add(point);
      // Pop out the root element if the max heap size is larger than K
      if (maxHeap.size() > K) {
        maxHeap.poll();
      }
    }

    return maxHeap.toArray(new int[0][]);
  }
}

The time complexity of this algorithm is $O(NlogK)$, and the space complexity is $O(K)$.

Conclusion

Ultimately, we have seen two kinds of top K elements problems solving by min heap and max heap, respectively. In summary, the top K elements pattern can be divided into two steps:

  1. Decide which kind of heap you’d like to use. Be careful of the heap initialization, especially the way to compare between elements.
  2. Maintain the heap size smaller than K. If it is larger than K, extract the root element. Repeat this process until the end.
